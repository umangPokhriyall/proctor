# proctor — crypto-overhead microbench methodology (Phase 2 §8)

This note records how the figures in this directory were produced and states each
result with its units and conditions. Numbers come from the CSVs here, regenerated
by `crypto/examples/crypto_overhead.rs`; prose never restates a number the CSV does
not contain. Distributions are reported, not just means.

## How to regenerate

```sh
cargo run --release --example crypto_overhead
```

The example segments the corpus, runs the four measurements, and overwrites the
CSVs in this directory. It requires ffmpeg; if ffmpeg is absent it prints a loud
skip and writes nothing (numbers are never fabricated).

## Host and toolchain (conditions)

| Field    | Value |
| -------- | ----- |
| CPU      | 11th Gen Intel Core i5-1135G7 @ 2.40 GHz (Tiger Lake), 8 logical cores |
| Kernel   | Linux 7.0.0-22-generic x86_64 |
| ffmpeg   | 8.0.1-3ubuntu2 |
| rustc    | 1.95.0, `--release` (opt-level 3) |
| AES-NI   | `aes=true`, `pclmulqdq=true` (runtime cpuid) |
| date     | 2026-06-11 (UTC) |

Single host, single thread of measurement, no core pinning, on a laptop CPU under a
normal desktop — treat absolute latencies as indicative, not datacenter-grade. The
ratios (crypto-as-%-of-transcode, memfd-vs-disk delta) are the load-bearing results
and are robust to host speed because both terms scale together.

## Corpus and segmentation

The committed seed corpus (`bench/corpus/`) is 320×240, 4 s, 30 fps, GOP=30
(`gradient` = testsrc2, `detail` = mandelbrot, `motion` = testsrc2+rotate+noise).
Each clip is split with `ffmpeg -c copy -f segment -segment_time 2` into ≈2 s
GOP-aligned segments — **6 segments**, 55 KB–1.86 MB. The on-host full set is
generated by raising `SIZE`/`DUR` via `corpus/generate.sh`; the same bench applies
unchanged. Profile steps are two representative downscales from 320×240:
`240×180 @ 400 kbps` and `160×120 @ 200 kbps`, both H.264/MP4.

## Results

### 1. AEAD throughput — `aead_throughput.csv`

Single-thread AES-256-GCM over the segment buffers (fresh 96-bit nonce per encrypt;
each `decrypt` returns plaintext into an `mlock`'d, zeroizing `SecretBuf`). Per
segment, ≥256 MB is pushed through so the timer is stable. Aggregate over all
segments:

- **encrypt: 1.55 GB/s** (1.61 GiB processed in 1.04 s)
- **decrypt: 0.99 GB/s** (1.61 GiB processed in 1.62 s)

Throughput is flat across segment sizes (1.43–1.56 GB/s encrypt, 0.95–1.01 GB/s
decrypt), i.e. it is per-byte cost, not per-call setup. **AES-NI is engaged:** the
`aes` crate (via `aes-gcm` 0.10) runtime-detects AES-NI, and cpuid reports
`aes`/`pclmulqdq` present on this CPU.

**What surprised us:** decrypt is ~36% slower than encrypt. The cause is by design,
not the cipher — `decrypt` pins its output plaintext with an `mlock` syscall (and
`munlock` on drop) so plaintext cannot swap, whereas `encrypt` writes ciphertext to
an ordinary `Vec`. The asymmetry is the measured cost of keeping plaintext out of
swap; we keep it rather than hide it.

**Forced-software comparison: not run.** Disabling AES-NI requires building the
`aes` crate with its `force-soft` feature, a separate compile-time configuration
this single example binary cannot toggle at runtime. We report the achieved
hardware throughput and state plainly that the software comparison was not measured,
rather than estimate it.

### 2. Per-segment crypto latency — `crypto_latency_raw.csv`, `crypto_latency_summary.csv`

For each (profile, segment), 2000 samples of `decrypt(source) + encrypt(output)`,
where `output` is that profile's transcoded plaintext. Raw per-sample µs are in
`crypto_latency_raw.csv` (24 000 rows); per-group and per-profile percentiles in
`crypto_latency_summary.csv`. Selected p50 / p99 / p99.9 (µs), profile `240×180@400k`:

| segment      | bytes  | p50 µs  | p99 µs  | p99.9 µs |
| ------------ | ------ | ------- | ------- | -------- |
| gradient_001 | 55 KB  | 62.6    | 77.2    | 88.7     |
| detail_000   | 439 KB | 401.1   | 484.9   | 533.7    |
| motion_000   | 1.86 MB| 1399.2  | 1749.0  | 1912.1   |

Latency tracks segment size, as expected for a single-shot AEAD. Distributions are
tight (p99.9 within ~1.4× of p50), consistent with a CPU-bound, allocation-light
path and no I/O.

### 3. Crypto as a percentage of transcode — `crypto_pct_transcode.csv`

Per (profile, segment): median ffmpeg `transcode_no_disk` wall time (5 reps) vs the
p50 decrypt+encrypt time.

- **Range: 0.10 % – 1.03 %** of transcode time.
- Lowest: `gradient_001` 0.10 % (240×180). Highest: `motion_000` 1.03 % (160×120) —
  the largest segment against the cheapest transcode.

The result is the intended one, stated as measured: at this confidentiality bound,
crypto costs **about a tenth of a percent to one percent** of the encode. It is
nearly free relative to the transcode.

### 4. memfd (anonymous RAM) vs naive disk path — `memfd_vs_disk.csv`

End-to-end `decrypt → transcode → encrypt`, median of 5 reps. The **memfd** arm is
the library path (`transcode_no_disk` over `/proc/self/fd/N`); the **disk** arm is
the legacy baseline — it `decrypt`s, writes the plaintext to a file, runs ffmpeg
from that file, then encrypts. Both run ffmpeg once with identical args; they differ
only in where plaintext lives.

- **Delta (disk − memfd): −4.2 ms to +7.5 ms**, centered near zero — i.e. within
  run-to-run noise. The anonymous-RAM path is **not** measurably slower than writing
  plaintext to disk.
- **Only the memfd arm passes the §7 no-disk proof.** The disk arm opens a
  regular-file fd holding the plaintext segment (`memfd_passes_no_disk=yes`,
  `disk_passes_no_disk=no`); see `bench/results/crypto/no-disk-audit.txt`.

So the confidentiality fix is free: it converts the legacy disk write into an equal-
cost memfd write while removing the plaintext-on-disk surface.

**What surprised us (and a fix it prompted):** the first run showed the memfd arm
2–24 ms *slower* than disk. That was not memfd I/O — it was `transcode_no_disk`'s
child-wait loop polling at 20 ms granularity and oversleeping past ffmpeg's exit.
The benchmark surfaced a real latency bug; the poll interval was tightened to 1 ms
(bounding overshoot to ~1 ms), after which the delta collapsed to the near-zero
figures above. Measure, never guess.
