# proctor — detection-probability family (Phase 3 §4)

This note records how the detection-probability artifacts in this directory were
produced and states the result with its conditions. Numbers come from the CSVs here,
regenerated by `verify/examples/detection_curves.rs`; prose never restates a number the
CSV does not contain.

## How to regenerate

```sh
cargo run --release --example detection_curves
```

Pure arithmetic — no ffmpeg, no corpus, no randomness — so it always runs and the CSVs
are byte-stable.

## What is published

The verifier samples `k = ⌈p·n⌉` of a job's `n` segments **without replacement** and
re-checks them. With `m = ⌈f·n⌉` segments tampered, the exact probability that at least
one tampered segment is caught is the **hypergeometric**

```
P_detect(f, n; p) = 1 − C(n−m, k) / C(n, k)
```

computed exactly (integer-exact for `n ≤ 64`) as the product `∏_{i=0}^{k−1} (n−m−i)/(n−i)`
— no Gamma function, no stats crate.

| File | Contents |
| ---- | -------- |
| `detection-family.csv` | The **published** hypergeometric family `P_detect(f, n; p_tier)` over the tiers, the segment grid, and the tamper grid. Columns: `tier,p,n,f,m,k,p_detect_hypergeometric`. |
| `detection-divergence.csv` | Both curves and `divergence = binomial − hypergeometric`. Columns: `tier,p,n,f,m,k,binomial,hypergeometric,divergence_binomial_minus_hyper`. |

**Grids.** Tiers `p ∈ {pristine = P_MIN = 0.02, mid = 0.10, high = 0.25}` (representative
reputation tiers; the tier→`p` *policy* is Phase 4, not Phase 3). Segments
`n ∈ {4, 8, 16, 32, 64}`. Tamper fraction `f ∈ {1/32, 1/16, 1/8, 3/16, 1/4, 3/8, 1/2, 3/4, 1}`.

## The `P_MIN` floor (amendment §1.3)

`P_MIN = 0.02` is a **hard floor** applied to every worker, pristine ones included, so
`k = ⌈P_MIN·n⌉ ≥ 1` across the whole grid (`n = 4 → k = 1`, `n = 64 → k = 2`): no worker
is ever unsampled, and there is always a minimum detection probability. The accepted leak
— a worker can infer its tier from its observed challenge rate — is tolerated *because* the
floor guarantees this minimum regardless of what the worker infers.

## The binomial divergence — and an honest correction of its sign

The legacy brief published the **binomial** `1 − (1−p)^(⌈f·n⌉)` (the with-replacement /
Bernoulli-sampling model). We keep it **only** for the divergence plot; the exact
hypergeometric is the published claim.

The amendment (§1.2.1) motivates this as "the binomial *overstates* detection at small `n`."
The committed data shows the divergence has the **opposite sign**: `binomial − hypergeometric
≤ 0` everywhere on the grid (`detection-divergence.csv`), so the **binomial under-states**
true detection. The gap is largest at small `n` — up to `−0.922` at `n = 4, f = 1, p = 0.02`
(binomial `0.078` vs. exact `1.000`), because the `k = ⌈p·n⌉` ceiling lifts the realized
sample fraction far above `p` there.

This is a rigorous fact, not a measurement artifact: for the "detect ≥ 1 tampered" event,
`P(miss)_hyper = ∏_{j=0}^{m−1} (n−k−j)/(n−j)`, and each factor `(n−k−j)/(n−j) ≤ (n−k)/n ≤ 1−p`
(first step because `(n−k−j)·n − (n−k)(n−j) = −kj ≤ 0`; second because `k = ⌈p·n⌉ ≥ p·n`), so
`P(miss)_hyper ≤ (1−p)^m = P(miss)_binomial` and hence `P_detect_hyper ≥ P_detect_binomial`.

The amendment's load-bearing **decision** is unchanged: publish the **exact** hypergeometric
and keep the binomial only to exhibit its approximation error. Only the *sign* of that error
is corrected here — by the proof and by the CSV, in keeping with the repo's measure-never-guess
and no-fabricated-claims rules. (Flagged for the spec owner: amendment §1.2.1's "overstates"
should read "under-states.")
